Distributed by design

Three datacenters, one platform

Each DC sits on its own ground plane — racks of physical hosts, each running a stack of microVMs. The dashed cyan threads are the WireGuard overlay : every microVM peers with every other across the WAN, so tenant workloads (sky-blue), infrastructure microVMs (indigo : etcd, nats, dex, zot, weft, weft-network, otel-collector, victoriametrics, perses), and CubeFS storage microVMs (teal : S3 + POSIX volumes) share the same flat address space regardless of which AZ they land in.

Quick start

What you can do today

# Run as the platform's control plane (single host or cluster node)
$ weft agent

# From another terminal — single host or any cluster client:
$ weft project create team-alpha
$ weft microvm run alpine:3.21 --project team-alpha -d

# Stream live events for one VM
$ weft events --vm web-1

# Bring up the infrastructure services (etcd, dex, zot, nats, cubefs, otel) in dependency order
$ weft infra bootstrap
$ weft infra status

The CLI talks to a running weft agent over gRPC ; on a Mac laptop the same binary plays both roles, so there is no separate daemon to install.

weft microvm is the default execution path : an OCI image becomes a fast-boot micro-VM sharing the substrate kernel, with its rootfs served over virtio-fs (Apple-VZ) or virtio-9p (QEMU). All new platform features land here first — Longhorn volumes, GPU flavors, CubeFS shares, mesh networking, observability. The classic full-VM path (weft instance) stays around as an escape hatch for the workloads microVMs can't reach : Windows / BSD guests, network appliances distributed as VM images (VyOS, OPNsense, …), and workloads that need their own kernel (forensics, KASAN, real-time). Same hypervisor drivers underneath, same scheduler, same flavors — only the boot model differs.

Multi-DC endpoints

API access

Each DC runs one weft agent microVM that exposes the gRPC API on a stable address. Clients resolve them via SRV records served by the per-DC CoreDNS microVMs and retry against the next server on failure — the same pattern as Consul or Nomad. No external load-balancer, no anycast, no extra component to operate. 

# SRV record served by CoreDNS, one entry per DC
_weft._tcp.weft.internal.  IN SRV  0 33 7443 weft-a.weft.internal.
_weft._tcp.weft.internal.  IN SRV  0 33 7443 weft-b.weft.internal.
_weft._tcp.weft.internal.  IN SRV  0 33 7443 weft-c.weft.internal.

# Client picks any agent ; reads are served locally,
# writes are forwarded to the etcd Raft leader transparently.
$ weft --servers _weft._tcp.weft.internal vm list
  • Stateless auth — every agent caches dex's JWKs, so OIDC token validation never crosses a DC boundary.
  • Transparent write routing — the contacted agent proposes via etcd Raft ; clients never have to know which DC holds the leader.
  • Public ingress, same shape — multi-A DNS resolves api.weft.example.com to one public IP per DC ; the client retries the next on failure.
  • Stickiness for streams — long-lived watches (weft events) stay pinned to the agent that accepted them ; SRV reshuffles only on reconnect.
Desktop & mobile

Client apps

The dashboard is a thin native shell around the same weft-webui the browser serves — one WebView, no duplicated UI. On the desktop it lives as a menu-bar / system-tray icon (macOS, Linux/GTK, Windows) ; on mobile it is a full-screen app (Android, iOS). Click the tray icon, the dashboard opens in a window ; everything you can do in the browser, you can do here.

weft-app-osx

macOS menu-bar app. Go + webview (WKWebView) + a status-bar item. Ships as a notarized .app bundle.

weft-app-gtk

Linux tray app. Go + webview (WebKitGTK) + StatusNotifierItem. Ships as a Flatpak.

weft-app-windows

Windows notification-area app. Go + webview (WebView2) + a tray icon. Ships as an MSIX.

weft-app-android

Kotlin + WebView. The failover supervisor runs in a foreground service.

weft-app-ios

Swift + WKWebView. The failover supervisor runs in the app process.

weft-app-core

Shared Go library — the transport-aware, anti-flap HA failover supervisor the three desktop apps embed. Mobile mirrors its logic natively.

Highly available by construction

A DC can go down without the operator noticing. Each app knows about every DC's weft-webui (resolved from the same per-DC SRV / multi-A DNS the API uses) and keeps a live health check on each. The dashboard's API client is handed the full endpoint list ; calls retry against the next healthy DC the instant one stops answering. Because the SPA assets are already in the WebView, a DC loss only re-routes the API traffic — the page never reloads, so the switch is invisible in the common case.

  • Seamless first, visible only if it must be — in-flight and subsequent calls fail over silently. If every known DC is unreachable, or a hard reload is unavoidable, a non-blocking banner says "Connection switched to DC-B" rather than leaving a dead page.
  • Anti-flap — the supervisor uses hysteresis : a DC must pass health checks for a hold-down window before it is promoted back, so a DC that is flapping up/down doesn't whipsaw the active connection.
  • Stickiness for streams — long-lived SSE event streams re-attach to the newly-active DC on failover, mirroring the weft events reconnection behaviour.

No worldwide web listener

Because the dashboard reaches the cluster through the app, the platform never has to expose weft-webui to the public Internet. The app brings its own authenticated transport and points the WebView at a private, loopback-only address ; only enrolled users and devices can reach the UI at all.

  • WireGuard transport — the app joins the cluster WireGuard mesh as a userspace peer (wireguard-go netstack, no tun privilege) and talks to each DC's weft-webui on its mesh address. The webui listener binds the mesh interface only.
  • SSH transport — zero-config fallback : the app opens an SSH connection to a per-DC endpoint and local-forwards a loopback port to the internal weft-webui (pure-Go x/crypto/ssh, no VPN entitlement — ideal on mobile and locked-down networks). One forward per DC means the SPA still sees a full endpoint list to fail over across.
  • Identity, end to end — the transport key (WireGuard or SSH) gates the network ; dex OIDC still gates the session inside the UI. Losing a device means revoking one key, not rotating a public secret.
Programmable network

Routers & load balancers

Tenant workloads need more than a flat WireGuard mesh : the platform exposes routers (mesh ↔ mesh peering, or mesh ↔ public Internet egress) and load balancers (L4 / L7 in front of microVMs). Both are first-class resources : declare them in HCL or via the dashboard, the data plane reconciles.

  • Data plane, Go-native by default — L7 (HTTP) via Caddy (Apache 2.0, pure-Go, auto-HTTPS via ACME) embedded inside weft-agent on every host. L4 (TCP/UDP) via the same Caddy through the layer4 plugin — one binary, no separate Envoy microVM. Mesh peering via WireGuard in-kernel. BGP egress, when a tenant needs a public ASN, lives in a small weft-router microVM running GoBGP (Apache 2.0, pure-Go, BGP-4 + EVPN + flowspec — the same engine Calico and Cilium ship) ; it programs the kernel FIB via netlink, boots in a few hundred milliseconds, footprint ~tens of MB. NAT egress for tenants without an ASN is plain netfilter / nft pushed by weft-network — no VM at all. Escape hatch : when a tenant brings a complex multi-protocol setup (OSPF / IS-IS / RSVP-TE) or wants its own VyOS / OPNsense config, run it as a classic VM via weft instance ; same deal as for Windows guests and other VM-image appliances.
  • Control plane — weft-network, a thin Go binary that subscribes to weft's event stream (LoadBalancer, Router, Network, microVM) and pushes deltas to each host's local Caddy via the admin API (POST /load on a unix socket owned by weft-agent). Three replicas, leader elected via etcd ; sub-second failover. The source-of-truth is the weft API, not a separate xDS bus.
  • Multi-DC HA — every host runs its own Caddy ; clients reach the platform through multi-A or SRV DNS (one record per DC) and retry the next on failure. Same shape the weft agent endpoints already use — no anycast, no external load balancer, no per-DC LB pool to operate.
Observability

Telemetry

Every weft component speaks OpenTelemetry. Control-plane daemons (weft agent, drivers, weft-webui) export metrics on a Prometheus-shaped /metrics endpoint ; future work adds traces and logs via the OTLP gRPC exporter.

The collection plane runs as three infra microVMs alongside the rest of the platform :

  • otel-collector — one replica per AZ. Scrapes the admin port of every weft daemon and webui (over the WireGuard mesh, so the public listener never exposes the TSDB). Aggregates, filters PII, then ships everything to the metric store.
  • victoriametrics — long-term Prometheus-compatible storage (CNCF sandbox). Single-binary, low-overhead, scales horizontally when needed. PromQL on the read side, so existing dashboards transfer over.
  • perses — dashboards as code (CNCF sandbox), SSO via the cluster's dex microVM so operators sign in with the same OIDC identity they use for the CLI and webui. Dashboards live in HCL alongside the rest of the cluster config ; alerting routes through Alertmanager.

The admin-facing weft-webui ports expose per-user activity counters (weft_webui_user_actions_total{sub,action}, weft_webui_auth_logins_total{result}) alongside the usual HTTP / gRPC histograms — so an operator can answer "who pushed which artifact, when ?" from the dashboard rather than grepping logs.

Compute & storage

Flavors, volumes & shares

Three primitives, deliberately separate. A flavor is the compute envelope — vCPU, RAM, optional GPU(s), and an optional cap on ephemeral scratch. A volume is single-attach block storage. A share is a CubeFS-backed POSIX filesystem that many microVMs mount at once (read-write-many). Keeping them decoupled is what lets a flavor stay reusable across very different workloads. 

# Flavor — compute envelope (vCPU, RAM, optional GPUs, optional ephemeral scratch)
$ weft flavor set small  --vcpu 2  --ram 4Gi  --ephemeral 8Gi

# GPU flavors — declared by model and count, scheduler matches free hardware
$ weft flavor set ai-h200 --vcpu 32 --ram 256Gi --gpu 1 --gpu-type nvidia-h200          --ephemeral 400Gi
$ weft flavor set ws-rtx6 --vcpu 16 --ram 96Gi  --gpu 1 --gpu-type nvidia-rtx-6000-ada  --ephemeral 200Gi

# Volume — single-attach block storage : default backend is Longhorn (replicated, snapshots, backups)
$ weft volume create pg-data --type block --size 100Gi --project team-alpha

# Escape hatch — pass a host device straight through (no replication, raw bandwidth)
$ weft volume create pg-fast --type block --source /dev/nvme1n1 --project team-alpha

# Share — a CubeFS POSIX filesystem several microVMs mount concurrently (RWX)
# The cubefs-storage plugin (Apache 2.0) is the default backend ; ceph or
# garage can replace it without re-mounting microVMs.

# Boot a microVM from an OCI image ; the dashboard / API wires
# the flavor + volume + share into the run request.
$ weft microvm run postgres:16 --project team-alpha
$ weft microvm run vllm/vllm-openai:latest --flavor ai-h200 --project team-alpha
  • No root disk to size — the rootfs is the OCI image itself (weft-microvm-init pivot-roots it over virtio-fs) plus a thin writable layer. So a flavor only caps the ephemeral scratch ; there is no persistent root volume to provision.
  • GPU as a flavor dimension — flavors can declare one or more GPUs by model (--gpu 2 --gpu-type nvidia-h200). The scheduler treats them like vCPU/RAM : it picks a host with that many matching cards free, then the hypervisor driver binds them at start time — VFIO PCI passthrough on the QEMU/KVM driver. NVIDIA MIG slices on the datacenter SKUs (H200, …) surface as just another GPU type from the scheduler's point of view ; workstation cards like the RTX 6000 Ada don't expose MIG, so the flavor consumes a whole device. The Apple-VZ driver doesn't expose discrete GPUs to guests, so GPU flavors are intentionally host-feature-gated ; scheduling them on a VZ-only host fails the placement up front rather than at boot.
  • Volume — single-attach block (RWO) — sized or sourced at creation and bound to one microVM, the way an instance type and a block volume are separate in the public clouds. The default backend is Longhorn (CNCF graduated, Apache 2.0) : replicated block storage with snapshots and backups spread across the host pool, so losing a host doesn't lose the volume. Two escape hatches stay available for specialised workloads — block passes a host device (NVMe / partition) straight through (raw bandwidth, no replication), and file is a host-side image. All three surface as virtio-blk via the weft Volume driver.
  • Share — multi-attach POSIX (RWX) — a CubeFS filesystem mounted concurrently by many microVMs, the way EFS or Filestore work : ideal for JupyterHub notebooks over a common dataset, or any fan-out of workers on shared data. The same bytes are reachable as objects through CubeFS's S3 endpoint.
  • Storage workloads, illustrated — a CubeFS DataNode is a modest flavor plus a large pass-through block volume ; a MetaNode is a RAM-heavy flavor with almost no disk. Same primitives, different mix.
Architecture

Design

  • Multi-DC control plane — three weft control-plane instances form an etcd quorum (one per AZ). Losing one DC keeps the platform writable ; the dispatch table re-routes around the failed host pool.
  • Multi-hypervisor — drivers load via go-plugin. Apple's Virtualization framework lands first (weft-driver-vz) ; QEMU/KVM (weft-driver-qemu) and OpenBSD's vmd (weft-driver-vmd) are the next siblings.
  • microVM-first, classic VM as escape hatch — weft microvm is the default execution path : OCI image → fast-boot micro-VM with shared kernel and virtio-fs / 9p rootfs. All new platform features land here first. weft instance covers the cases microVMs can't reach — Windows / BSD guests, network appliances (VyOS, OPNsense, …) distributed as VM images, and workloads that need their own kernel (forensics, KASAN, real-time). Both modes share the same hypervisor drivers, scheduler, placement rules, flavors (vCPU / RAM / GPU), volumes and shares — only the boot model differs. We don't invest new features into the classic-VM path ; it exists so an operator never hits a wall.
  • Multi-tenant — every resource is UUID-keyed and owned by a project + user. ACLs are first-class : OIDC tokens from dex carry tenant grants ; weft-client threads them onto every gRPC call.
  • Multi-AZ + multi-rack placement — HA intent expressed declaratively. A 3-replica NATS cluster declares placement { count = 3, az = "different", rack = "different", host = "different" } ; the scheduler picks one host per replica honouring the proximity hierarchy AZ ⊃ Rack ⊃ Host.
  • Infrastructure as micro-VMs — etcd, dex, zot, NATS, CoreDNS, Longhorn (distributed block storage : replicated volumes, snapshots, backups), CubeFS (distributed POSIX + S3 shares), the networking control plane (weft-network), and the observability stack (otel-collector / victoriametrics / perses) all run as weft-microvm-init micro-VMs on the same substrate as user workloads. The L4/L7 data plane is Caddy embedded directly in weft-agent on every host — no separate proxy microVM. Every component is a libre-licensed project (Apache 2.0 / BSD / LGPL) — CNCF-rooted where applicable (etcd, CoreDNS, CubeFS & Longhorn graduated, NATS & OpenTelemetry incubating, dex, zot, VictoriaMetrics & Perses sandbox) — so the platform stands entirely on vendor-neutral, cloud-native foundations. One management plane, weft start / stop / status works for everything.
  • HCL over JSON for configs and plans. Comments, references, expressive types. JSON only for append-only log streams.
  • Go-native, no C dependencies in the control plane. Drivers can use cgo where the framework needs it (Apple-VZ does) ; the daemon itself doesn't.
Modules

Components

weft

Single binary — control daemon (weft agent) + CLI subcommands (weft microvm, weft project, …). Embeds Caddy as the L4/L7 data plane.

weft-proto

gRPC contract and generated stubs. Stable wire boundary between the daemon, the CLI, and the drivers.

weft-driver-plugin

go-plugin contract for hypervisor drivers. Each driver ships as a standalone subprocess speaking this gRPC schema.

weft-driver-vz

Driver for Apple's Virtualization framework. Boots VMs on macOS via the Code-Hex/vz bindings.

weft-driver-qemu

QEMU/KVM driver — Linux hosts ; also doubles as the macOS dev backend for nested-virt-free testing.

weft-driver-vmd

OpenBSD vmd driver — shells out to vmctl(8) for the lifecycle. BSD escape-hatch host class ; pure-Go, CGO=0.

weft-network

Networking control plane — reconciles Routers, Load Balancers, DNS zones and Scheduling Rules from etcd into the Caddy data plane and the per-tenant weft-router microVMs. One process per DC, leader-elected.

weft-router

Per-tenant BGP speaker + FIB programmer. Go-native micro-VM running GoBGP for BGP-4 + EVPN + flowspec peering with the upstream ISP ; installs received routes in the kernel FIB via netlink.

weft-block

Block-storage data plane — fork-and-adapt of longhorn-engine with a Go-native control plane and NBD frontend. Pure-Go, builds linux/arm64 with CGO=0. Default backend for weft volume.

weft-client

Shared dial / token / event-stream helpers consumed by the weft binary and the Terraform provider.

weft-webui

Browser dashboard — huma (Go) + Svelte 5 / daisyUI. Same gRPC API surface as the CLI ; mono-binary deploy.

weft-microvm-init

Tenant micro-VM init — minimal PID 1 that pivot-roots an OCI image and runs it through an embedded crun.

weft-microvm-agent

In-guest companion daemon — applies dynamic config from NATS (WireGuard mesh, SFTP/FUSE mounts, SSH keys) and runs first-boot provisioning (git via go-git, OCI via oras-go).

weft-microvm-kernel

Custom Linux kernel for the microVM substrate — minimal config, WireGuard built-in, 9p/virtio-fs, distributed as an OCI artifact.

terraform-provider-weft

Terraform provider — every resource declared as HCL maps to a weft API object.

weft-app-core

Shared Go library for the desktop apps — transport-aware (WireGuard / SSH / direct), anti-flap multi-DC failover supervisor + WebView & tray helpers.

weft-app-{osx,gtk,windows}

Desktop menu-bar / tray apps. Go + webview hosting weft-webui ; embed weft-app-core for HA failover.

weft-app-{android,ios}

Mobile dashboard apps. Kotlin WebView / Swift WKWebView ; mirror the failover supervisor natively.
